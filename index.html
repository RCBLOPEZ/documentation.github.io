01 Philosophy

Architecture Philosophy

Our platform is structured as a data lake with a medallion-like architecture — a widely adopted pattern where data is progressively cleansed, enriched, and refined as it moves through distinct layers. We combine this with Data Mesh principles to define ownership and standards on top of the physical structure.

Medallion Pattern — How Our Zones Map

Medallion LayerOur ZoneGCP ProjectGitHub RepoPurpose
Bronze Raw Zone project-raw dbt-raw Land data exactly as it arrives. Immutable. No transformations.
Silver Trusted Zone project-trusted dbt-trusted Cleanse, standardize, and structurally enrich. Safe to build on.
Gold Refined Zone project-refined dbt-refined Apply business rules. Produce consumable data products for reporting and analytics.

Each zone is a distinct GCP project and a distinct GitHub repository. This is an intentional boundary — access control, CI pipelines, and Cloud Run deploy triggers are fully isolated per zone. Cross-zone references are resolved via BigQuery dataset references between projects, not internal DBT refs within a single repo.

Data Mesh Principles

◈ Data as a Product
  • →Every model has a defined consumer
  • →Clear interface, tests, and documentation
  • →Grain always explicit — one row per what
  • →Accountable engineer for each product
◈ Federated Governance
  • →Standards agreed collectively
  • →Applied consistently across all zones
  • →This guide is that agreement
  • →Evolves as the team grows

03 Deep Dive

Trusted Zone — In Depth

The Trusted Zone (Silver layer) is where raw data becomes reliable. It contains exactly two DBT layers: staging and intermediate. Both live in dbt-trusted and run against project-trusted.

When to use stg_
One model per source table, always. Rename columns, cast types, apply basic null handling. Nothing else.
  • Column renaming and type casting
  • Removing obvious duplicates that arrive from the source
  • Basic null coalescing on known-noisy fields
  • Never: joins, aggregations, business logic, or filtering beyond deduplication
models/staging/stg_orders.sql — dbt-trusted 
-- Materialization: view  |  Repo: dbt-trusted

with source as (
    select * from {{ source('raw_operations', 'orders') }}
),

renamed as (
    select
        order_id::string          as order_id,
        customer_id::string       as customer_id,
        order_date::date          as order_date,
        status::string            as status,
        total_amount_cents::int   as total_amount_cents,
        coalesce(currency, 'GBP') as currency,
        _loaded_at               as loaded_at
    from source
)

select * from renamed
When to use int_ in Trusted Zone
Structural logic only — joining, enriching, and building reusable datasets. The key distinction from Refined int_ is intent: Trusted intermediates are about data structure and quality, not business meaning.
  • Joining two or more stg_ models together
  • Enriching records with reference data or lookups
  • Building a reusable dataset referenced by multiple Refined models
  • Never: business KPIs, revenue calculations, business rule filters, or metric definitions
models/intermediate/int_orders_with_customer.sql — dbt-trusted 
-- Materialization: view  |  Repo: dbt-trusted
-- Structural join only — no business logic

with orders as (
    select * from {{ ref('stg_orders') }}
),

customers as (
    select * from {{ ref('stg_customers') }}
),

joined as (
    select
        o.order_id,
        o.order_date,
        o.status,
        o.total_amount_cents,
        o.currency,
        c.customer_id,
        c.customer_name,
        c.country_code
    from orders o
    left join customers c on o.customer_id = c.customer_id
)

select * from joined

04 Deep Dive

Refined Zone — In Depth

The Refined Zone (Gold layer) is where data becomes a product. It contains three DBT layers: intermediate (business logic), facts, and dimensions. All three live in dbt-refined and run against project-refined.

When to use int_ in Refined Zone
This is where business rules live. Refined intermediates transform structurally-sound trusted data into business-meaningful outputs. They are ephemeral because they exist to power mart outputs, not to be queried directly.
  • Applying business rules: which orders count as revenue, what defines a confirmed customer
  • Calculating KPIs: revenue, margin, conversion rate, churn logic
  • Complex aggregations that power multiple fact tables
  • Shared logic reused across multiple fct_ or dim_ models
models/intermediate/int_revenue_recognized.sql — dbt-refined 
-- Materialization: ephemeral  |  Repo: dbt-refined
-- Business rule: revenue recognized on confirmed + shipped orders only, in GBP

with orders as (
    -- Cross-project: source() required — dbt cross-project ref not supported
    select * from {{ source('trusted', 'int_orders_with_customer') }}
),

fx as (
    select * from {{ source('trusted', 'stg_exchange_rates') }}
),

recognized as (
    select
        o.order_id,
        o.order_date,
        o.customer_id,
        o.customer_name,
        o.country_code,
        -- Business rule: convert to GBP using daily rate
        round(o.total_amount_cents / 100.0 * fx.rate_to_gbp, 2) as revenue_gbp
    from orders o
    left join fx
        on  o.currency    = fx.from_currency
        and o.order_date = fx.rate_date
    where o.status in ('confirmed', 'shipped')
)

select * from recognized
When to use fct_
Fact models are the consumable outputs of the Refined Zone. They represent business events or measurements at a clearly defined grain. Every fact model must state its grain explicitly in the model description.
  • Representing a business event: a sale, a payment, a shipment
  • Aggregating metrics to a defined time grain: daily, weekly, monthly
  • Building the dataset that BI tools will query directly
  • Grain must be explicit in the description: "one row per calendar day"
models/marts/fct_daily_revenue.sql — dbt-refined 
-- Materialization: table  |  Repo: dbt-refined
-- Grain: one row per calendar day

with revenue as (
    select * from {{ ref('int_revenue_recognized') }}
),

daily as (
    select
        order_date                   as revenue_date,
        count(distinct order_id)     as order_count,
        sum(revenue_gbp)             as total_revenue_gbp,
        avg(revenue_gbp)             as avg_order_value_gbp
    from revenue
    group by order_date
)

select * from daily
When to use dim_
Dimension models represent business entities — the who, what, and where. They provide the descriptive context needed to filter and slice fact metrics in BI tools.
  • Representing a business entity: customer, product, date, geography
  • Providing descriptive attributes used to filter or group facts
  • One current row per entity — not a history or event table
  • Referenced via foreign key relationships from fct_ models
models/marts/dim_customer.sql — dbt-refined 
-- Materialization: table  |  Repo: dbt-refined
-- Grain: one row per customer (current record)

with customers as (
    -- Cross-project: use source() not ref() — dbt cross-project ref not supported
    select * from {{ source('trusted', 'stg_customers') }}
),

final as (
    select
        customer_id,
        customer_name,
        country_code,
        created_date,
        case
            when country_code in ('GB','IE')       then 'UK & Ireland'
            when country_code in ('DE','AT','CH') then 'DACH'
            else 'Other'
        end as region
    from customers
    where is_deleted = false
)

select * from final

07 Standards

Naming Conventions

Names encode zone, layer, and subject. You should be able to understand a model's purpose from its name alone without opening the file.

Zone Layer Prefix Example Materialization
Raw Source declaration — source('raw_finance', 'invoices') External / n/a
Trusted Staging stg_ stg_invoices, stg_orders view
Trusted Intermediate int_ int_orders_enriched view / ephemeral
Refined Intermediate int_ int_revenue_calc ephemeral
Refined Mart — Fact fct_ fct_daily_revenue table
Refined Mart — Dimension dim_ dim_customer, dim_product table

Zone Boundary Rules

✓ Correct patterns
  • stg_ refs source() only
  • trusted int_ refs stg_ and other trusted int_
  • refined int_ refs trusted int_ via source() (cross-project)
  • fct_/dim_ refs refined int_ via ref(), or trusted stg_ via source() (cross-project)
  • No mart may ref() another mart
✗ Never do these
  • Direct raw or trusted table reference (schema.table)
  • Mart referencing another mart
  • stg_ referencing an int_ model
  • ref() across GCP project boundaries — use source() instead
ref() and source() examples 
-- ✓ Staging: refs source() only (Raw Zone via BQ cross-repo)
select * from {{ source('raw_finance', 'invoices') }}

-- ✓ Trusted int_: refs stg_ models
select * from {{ ref('stg_invoices') }}

-- ✓ Refined int_: cross-project use source() not ref()
select * from {{ source('trusted', 'int_invoices_enriched') }}

-- ✓ Mart: refs refined int_ in same project
select * from {{ ref('int_revenue_recognized') }}

-- ✓ dim_: cross-project stg_ via source()
select * from {{ source('trusted', 'stg_customers') }}

-- ✗ Never: direct schema.table reference
select * from raw.finance_invoices

-- ✗ Never: ref() across GCP project boundaries
select * from {{ ref('int_invoices_enriched') }}  -- use source() instead

-- ✗ Never: mart referencing another mart
select * from {{ ref('fct_daily_revenue') }}  -- inside another mart
08 Standards

Materialization

Default materializations are set in dbt_project.yml and enforced by zone. Override only with a clear technical justification noted in your PR.

dbt_project.yml 
models:
  dbt_trusted:
    staging:
      +materialized: view       # Trusted Zone · stg_
      +tags: ['trusted', 'staging']
    intermediate:
      +materialized: view       # Trusted Zone · int_
      +tags: ['trusted', 'int']

  dbt_refined:
    intermediate:
      +materialized: ephemeral  # Refined Zone · int_ (business logic)
      +tags: ['refined', 'int']
    marts:
      +materialized: table      # Refined Zone · fct_ and dim_
      +tags: ['refined', 'mart']
Override guidance
Use incremental on large fact tables that grow continuously. Use table on trusted intermediates referenced heavily downstream. Always document the reason in the model config comment and PR description.
09 Standards

Testing

Tests are the contract between zones. Test coverage requirements increase as data moves toward the Refined Zone. Models without minimum tests cannot be merged.

TestRaw ZoneTrusted ZoneRefined Zone
source freshness ✅ Required — —
not_null — Required on PKs and critical FK columns Required on all key and non-nullable fields
unique — Required on all primary keys Required on all primary keys
accepted_values — Recommended on status / type columns Required on all status, type, category fields
relationships — Recommended on foreign keys Required on all foreign key columns
schema.yml — test examples 
models:
  - name: stg_invoices
    description: 'Staged invoices — Trusted Zone'
    columns:
      - name: invoice_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['draft', 'sent', 'paid', 'overdue', 'cancelled']

  - name: fct_daily_revenue
    description: 'Daily revenue by category — Refined Zone'
    columns:
      - name: revenue_date
        tests: [not_null]
      - name: customer_id
        tests:
          - relationships:
              to: ref('dim_customer')
              field: customer_id
10 Standards

Documentation

Documentation is part of the data product. Consumers of a Refined Zone model must be able to understand what it contains, how it is built, and who owns it without asking an engineer.

Trusted Zone — minimum
  • →Model description: what it represents and what cleaning was applied
  • →Key columns and non-obvious fields
  • →Known data quality issues or caveats
Refined Zone — required
  • →Full description with grain, filters, and business rules
  • →All columns described
  • →Exclusions, time zones, and currencies stated
Refined Zone model doc template 
- name: fct_daily_revenue
  description: >
    Daily revenue aggregated from confirmed and shipped orders.
    Grain: one row per calendar day per product category.
    Excludes: cancelled orders, refunded transactions, test accounts.
    Currency: all values in GBP.
    Time zone: UTC.
  config:
    tags: ['finance', 'refined', 'fct']
Good descriptions answer these questions
  • What is the grain of this model — one row per what?
  • What business process or source does it represent?
  • Are there filters, exclusions, or business rules applied?
  • What time zone and currency are used?

12 Workflow

CI Pipeline

Automated checks run on every pull request across all three repos. They enforce zone boundaries and test coverage so reviewers can focus on logic and design.

dbt-raw CI
  • →dbt compile
  • →dbt source freshness
  • →Fails if source exceeds error_after
dbt-trusted CI
  • →dbt compile
  • →dbt test --select state:modified+
  • →Tests on changed models and dependents
dbt-refined CI
  • →Full dbt test suite
  • →Documentation check
  • →Peer review required
